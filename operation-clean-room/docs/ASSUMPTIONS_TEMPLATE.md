# Assumptions Log

Log only the decisions that materially change outputs, interpretation, or edge-case behavior. If a choice affects ARR, reconciliation, customer health, or dashboard semantics, record it here.

## What To Capture

For each assumption, include:
1. The assumption
2. Why we chose it
3. What breaks if it is wrong
4. The decision date

## Metric Definitions

| # | Assumption | Rationale | Impact if Wrong | Date |
|---|-----------|-----------|----------------|------|
| 1 | Chargebee `mrr` should be computed as monthly recurring value from normalized plan price plus recurring addons, with annual subscriptions divided by 12 | The raw JSON does not provide a normalized `mrr` field, but it does provide plan price, billing interval, and addon amounts needed to derive it consistently for downstream metrics | ARR, NRR, and subscription-to-payment comparisons would be overstated for annual plans or understated if addons were excluded | 2026-04-15 |
| 3 | Non-USD Stripe and Legacy amounts should be converted into USD at ingestion time, while USD records should be left unchanged | The current normalized interfaces do not expose a separate USD amount field, but downstream reconciliation expects comparable amounts across currencies | Cross-currency reporting would be inconsistent if some records stayed in native units while others were compared in USD | 2026-04-15 |
| 4 | Salesforce `acv` should be derived from `amount * 12 / contract_term_months`, with `amount` treated as `tcv` | The real opportunities export provides `amount` and `contract_term_months` but not explicit `acv` / `tcv` columns, and multi-term deals need an annualized value | ARR and pipeline comparisons would be materially wrong for 24- or 36-month deals if `amount` were interpreted as annual value | 2026-04-15 |
| 5 | ARR should be calculated from Chargebee plus non-duplicative Stripe and Legacy recurring revenue, not Chargebee alone | The CFO brief explicitly calls out three billing systems and migration duplicates; Chargebee is the subscription source of record, but Stripe and Legacy can still represent active recurring revenue or historical billing coverage | ARR would be understated if active Stripe/Legacy recurring customers were omitted, or overstated if duplicated systems were simply summed | 2026-04-15 |
| 6 | When `/api/metrics/arr` has no `date` query, it should use the latest active billing snapshot date from Chargebee instead of the server's current date | The local dataset's active subscription terms are in early 2025; using the actual server date in 2026 returns zero ARR because no sample subscriptions are active then | Dashboard users would see an empty ARR response even though the sample dataset contains active subscriptions | 2026-04-15 |
| 7 | Monthly revenue summary should use month-end ARR snapshots from January 2024 through the latest active billing snapshot | The CFO brief asks for ARR by month back to January 2024, and month-end snapshots are the most defensible board presentation basis for recurring-revenue trend lines | Mid-month starts, cancellations, or upgrades are represented in the month-end state rather than weighted by active days inside the month | 2026-04-16 |
| 8 | The dashboard's "recognized revenue" line should be an MRR run-rate estimate, with timing exceptions flagged separately rather than implementing ASC 606 | The CFO brief later clarifies that full revenue recognition is next quarter and clean ARR is the board-meeting priority | Auditors should not treat the MRR run-rate as GAAP recognized revenue; timing flags identify records needing Finance review | 2026-04-16 |

| 10 | NRR, churn, and cohorts should use the same month-end ARR snapshots as the revenue summary | The assignment prioritizes defensible board numbers, and using one unified snapshot path prevents metrics from disagreeing because of separate source filters | Intra-month upgrades/cancellations are netted rather than shown as separate movement events | 2026-04-16 |
| 11 | Unit-economics LTV should floor monthly logo churn at 0.5% when the observed period has little or no churn | Short sample periods can produce zero observed churn, which would create infinite LTV and make the board view less useful | LTV may be understated for very sticky segments, but avoids overstating acquisition efficiency | 2026-04-16 |
| 12 | Board-facing ARR plan mix should use four canonical billing tiers: `Enterprise`, `Scale`, `Growth`, and `Starter` | The live billing data contains active `Scale` and `Growth` subscriptions as separate tiers, and collapsing them obscures real plan mix on the revenue dashboard | Board reporting would misstate the plan distribution and hide the active Scale base if those tiers were merged | 2026-04-16 |

## Business Logic

| # | Assumption | Rationale | Impact if Wrong | Date |
|---|-----------|-----------|----------------|------|
| 1 | Stripe `amount` values in `stripe_payments.csv` are already in the business reporting unit used by this dataset and should not be scaled by 100 during ingestion | The file contains small plan-like values such as `63`, `68`, and `-58`; dividing by 100 would produce implausibly tiny subscription payments for the sample dataset | Stripe revenue would be understated by 100x and any billing comparison against subscriptions would be unusable | 2026-04-15 |
| 3 | Chargebee subscriptions should accept either a top-level array or a `{ subscriptions: [...] }` wrapper, but this dataset's wrapper form is the source of truth | The real file is an object containing a `subscriptions` array, while future exports may reasonably flatten to an array; supporting both keeps ingestion resilient without changing the normalized contract | Ingestion would fail entirely if the payload wrapper differed slightly between exports | 2026-04-15 |
| 4 | Missing Chargebee customer name parts and billing address fields should be backfilled with safe defaults rather than rejected | The source JSON usually provides `company` and `email`, but often omits `first_name`, `last_name`, and full address details even though the normalized interface requires a richer customer shape | Overly strict validation would discard otherwise valid subscriptions and reduce match coverage across systems | 2026-04-15 |
| 5 | Salesforce owner emails should be synthesized from owner names because the exports do not include an email column | The account and opportunity CSVs include owner names but no owner email field, while the normalized interface requires one | Any workflow that relies on owner email could break or route incorrectly if a deterministic fallback were not provided | 2026-04-15 |
| 6 | Salesforce account segment should be inferred from employee count thresholds rather than source account status or subscription type | The export does not provide a direct segment column, but employee count offers a stable proxy suitable for the normalized interface | Segmentation analyses could shift if the business defines segment differently than the loader's size-based heuristic | 2026-04-15 |
| 7 | Salesforce `annual_contract_value` is the closest available source field for the normalized `annual_revenue` field in this dataset | The accounts export does not include true company revenue, only contract-value-like fields, so the normalized field is populated from the nearest business metric available | Downstream consumers may overinterpret `annual_revenue` as company financial revenue rather than CRM contract value | 2026-04-15 |
| 8 | Stripe subscriptions can be inferred from grouped recurring payments using payment cadence when no explicit Stripe subscription-state export exists | The Stripe dataset contains payments with `subscription_id` but no separate subscription lifecycle table, so ARR must infer active periods from payment dates and expected billing intervals | Stripe ARR may be over- or under-estimated if payments are irregular, skipped, retried, or not truly recurring | 2026-04-15 |
| 9 | Legacy recurring ARR should be treated as a fallback source only when not linked to Stripe and not already represented in active Chargebee or Stripe records | Legacy invoices are historical billing data and some rows cross-reference Stripe through `payment_ref`; direct summing would double-count migrated or linked revenue | ARR could be overstated if old-system invoices are counted alongside their newer billing-system equivalents | 2026-04-15 |
| 10 | ARR movement buckets are calculated from customer-level month-over-month ARR deltas | The available normalized sources provide active recurring values, but not a complete booked-event ledger across every billing system; customer-level deltas still produce an explainable board waterfall | A customer that leaves and returns within the same month may be summarized as a net movement rather than separate churn and reactivation events | 2026-04-16 |
| 11 | Customers first seen during a month are classified as new business; existing customers with no prior active month but older first-seen dates are classified as expansion/reactivation | This avoids presenting reactivated legacy or migrated records as brand-new logos when their source start date predates the current month | Some reactivations may be better labeled separately once a full lifecycle event model exists | 2026-04-16 |
| 12 | Marketing `attributed_deals` is the denominator for channel CAC | The marketing file provides channel spend and attributed deal counts but no customer-level multi-touch path; the CFO asked for a defensible interim choice rather than a full attribution rebuild | Channels with duplicate attribution may show distorted CAC; the dashboard should treat channel CAC as directional | 2026-04-16 |
| 13 | Pipeline zombie age is measured from the best available CRM activity proxy, falling back to close date after normalization | The raw Salesforce export includes `last_activity_date`, but the normalized opportunity type does not currently preserve it; close date is retained and stable | Some open opportunities may appear fresher or staler than they truly are until last-activity is added to the normalized model | 2026-04-16 |
| 14 | Reconciliation discrepancy resolution can be process-local for this emergency dashboard | The immediate requirement is a presentable board-readiness view; durable review workflow is not yet scaffolded in the monorepo | Resolved status is lost on API restart and is not sufficient for a formal audit signoff | 2026-04-16 |
| 15 | The discrepancies summary should show current billing exposure, not cumulative CRM pipeline ACV | The CFO's clean ARR/MRR view is a current billing number; adding historical closed-won CRM ACV mismatches made the top-line impact look larger than the revenue base and was not comparable | Pipeline mismatches still appear in the table, but users must inspect them separately from billing exposure | 2026-04-16 |
| 16 | A reconciliation run without an explicit date range should default to the latest billing snapshot month | The dashboard compares discrepancies against the current clean MRR view; using the full historical sample by default made the exposure number a cumulative backtest rather than a current close-period risk | Historical reconciliation still requires explicit `dateStart` and `dateEnd` inputs; otherwise older timing differences are intentionally excluded from the headline exposure | 2026-04-16 |
| 17 | Pipeline amount mismatches should not be aggregated into a dollar KPI until Salesforce amount semantics are cleaned | The inspected Salesforce opportunity and account exports contain amounts at materially different magnitudes from billing ARR; summing those deltas produced a large number that looked like a units bug and was not board-safe | Pipeline rows are retained as CRM review findings, but the dashboard reports a count of reviews instead of a summed pipeline ACV delta | 2026-04-16 |
| 18 | Pipeline comparisons should use active-term closed-won opportunities as of the ARR snapshot | Comparing every historical closed-won opportunity to current billing ARR mixed expired bookings with current revenue and inflated discrepancy volume | Expired contracts are excluded from current pipeline-vs-billing checks unless a user explicitly builds a historical reconciliation flow later | 2026-04-16 |
| 19 | Stripe recurring-plan inference should use the most recent recognizable subscription description in a payment stream, not the latest payment row blindly | Several active Stripe subscriptions end with refund, dispute, or generic payment descriptions even though earlier recurring rows still identify the actual plan tier | Active recurring Stripe customers can fall into `Unknown` and distort plan mix if inference trusts the last row regardless of description quality | 2026-04-16 |
| 20 | ARR should only suppress cross-system Stripe overlap when duplicate detection explicitly classifies the Stripe subscription as a `true_duplicate` | The current logic intentionally preserves possible multi-product or migration rows unless the duplicate classifier has high enough confidence; traced February 2025 overlaps like Lumin Digital, Novatech, and Horizon Media Partners show this is conservative but can still leave migration-style double counting in the board view | Clean ARR may remain overstated during billing cutovers until Finance adds customer-level migration mappings or a stricter cutover rule | 2026-04-17 |
| 21 | Reconciliation should compare CRM against the summed active billing footprint per normalized account, while still surfacing the contributing billing systems separately | Comparing Salesforce independently to Stripe, Chargebee, and Legacy created false negatives for accounts that correctly lived in only one current billing system; summing the active billing footprint answers the over-reporting / under-reporting question the CFO asked while preserving system-level drilldown in the discrepancy detail | If Finance instead wants CRM checked against a designated billing system of record per account, some current CRM mismatch rows would need to be reclassified once that mapping exists | 2026-04-17 |
| 22 | Billing-versus-CRM discrepancies should use a strict `2%` percent-delta threshold, and missing one side counts as a mismatch against zero | The requirement explicitly asks to flag anything beyond `2%` and says not to excuse rounding differences above that point; treating missing CRM or missing billing as zero makes those gaps visible in the same framework | Finance could see more discrepancies than before, especially around migrations and unmapped accounts, but that is preferable to silently hiding material board-readiness issues | 2026-04-17 |

## Exclusions & Edge Cases

| # | Assumption | Rationale | Impact if Wrong | Date |
|---|-----------|-----------|----------------|------|
| 1 | Empty Stripe optional fields such as `subscription_id`, `description`, `failure_code`, `refund_id`, and `dispute_id` should be normalized to `null` rather than left as empty strings | The CSV export represents missing values as blank cells, while the normalized TypeScript model uses nullable fields | Downstream business logic may treat empty strings as real identifiers or fail null checks designed around the shared type contract | 2026-04-15 |
| 3 | Only currently active Chargebee coupons with `apply_on = invoice_amount` should reduce the derived `mrr` during ingestion | The real dataset includes expiring coupons, perpetual coupons, and invoice-level discount metadata; applying only active invoice-wide discounts keeps the derived MRR tied to the current recurring term | Historical or item-level discounts could incorrectly suppress present-day MRR, leading to understated subscription value | 2026-04-15 |
| 4 | Legacy `payment_ref` values should be treated as nullable and trusted only when they match real Stripe-like identifiers | The inspected XML uses blank tags for most rows and Stripe-like `pi_...` values for a smaller subset; the loader does not attempt broader pattern interpretation beyond preserving the reference | Incorrect linkage between legacy invoices and Stripe payments could create false reconciliation matches | 2026-04-15 |
| 5 | Unsupported FX currencies should fail fast rather than silently passing through unconverted | The FX utility only has historical series for `USD`, `EUR`, `GBP`, `JPY`, and `AUD`, and silent passthrough would hide unsupported-currency errors | Financial metrics could be wrong without obvious failure signals if an unknown currency slipped through as if it were USD | 2026-04-15 |
| 6 | Timing issue flags should include annual Chargebee subscriptions, disputed/refunded Stripe payments, and overdue/partially paid legacy invoices | These are the clearest available indicators of timing or collectability risk in the current local files and match the CFO's examples of prepayments and not-yet-clean revenue | The list is a review queue, not an exhaustive rev-rec memo; Finance may add service-delivery or contract-liability rules later | 2026-04-16 |
| 7 | `plan_pricing_history.csv` should be treated as a reference dataset for pricing semantics, not an active input to ARR calculation in the current pipeline | The current ARR and revenue-summary path derives plan mix from live billing sources and does not load or join the pricing history file anywhere in the implementation | Future readers could assume pricing history is already enforcing plan taxonomy when it is only being used as a validation aid today | 2026-04-16 |
| 8 | Revenue dashboard charts should add a baseline zoom companion when one month is an extreme outlier, while leaving the full-scale chart intact | The February 2025 Chargebee snapshot is more than 20x the prior monthly ARR peak, which makes Jan 2024-Jan 2025 visually unreadable on a single linear axis; the CFO still needs to see the real spike, not a filtered dataset | If Finance prefers one chart only, the extra zoom panels may feel visually busy, but they avoid hiding the underlying data while preserving historical readability | 2026-04-17 |
